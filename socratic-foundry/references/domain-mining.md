# 경험 광산 + 암묵지 변환기 — mining phase 실행 지침

`mining` phase(Gate 4)에서 사용자의 경험을 파서 판단 규칙(`judgmentRules`)과
비대칭 자산(`assets`)으로 바꾸는 작업 지침이다. 사업 기회의 원석은 아이디어가 아니라
사용자가 이미 반복해서 해결해 온 문제 안에 있다.

---

## 1. 경험 광산 — 어디를 파는가

이력서의 경력 항목은 광맥이 아니다. 아래 7곳을 판다:

| 광맥 | 캐는 질문 |
|------|-----------|
| 반복 해결한 문제 | "지난 몇 년간 가장 여러 번 해결한 문제는 무엇입니까?" |
| 자주 맡는 역할 | "팀이나 모임에서 자연스럽게 당신에게 넘어오는 역할은?" |
| 남들은 어려워하지만 자연스러운 일 | "남들이 '어떻게 그걸 하냐'고 하는데 본인은 별것 아닌 일은?" |
| 실패를 미리 알아차리는 신호 | "일이 틀어지기 전에 먼저 느끼는 이상 신호가 있습니까?" |
| 타인이 반복 요청하는 도움 | "사람들이 뭘 들고 당신을 찾아옵니까?" |
| 자신만의 우회 방법 | "정석대로 안 하고 본인만의 방식으로 처리하는 일은?" |
| 오래 축적한 자료·사례 | "폴더·노트·스프레드시트에 몇 년째 쌓아온 것이 있습니까?" |

답변에서 나온 실제 사건은 `events[]`에, 반복 확인된 것은 `ledger.facts`에 기록한다.
자료·관계·신뢰 같은 축적물은 §5의 `assets[]`로 넘긴다.

---

## 2. 암묵지 변환기 — "감으로 안다"의 분해

사용자가 "그건 보면 안다", "감이다"라고 답하면 그 지점이 가장 값진 광맥이다. 다음 5개 프로브로 분해한다:

1. "어떤 **표현**을 들으면 '이상하다'고 느낍니까?"
2. "상대의 **말과 행동** 중 무엇을 더 신뢰합니까? 왜?"
3. "경험이 적은 사람이 같은 상황에서 **놓치는 신호**는 무엇입니까?"
4. "그 판단이 **빗나가는 경우**는 언제입니까?"
5. "이걸 신입에게 **설명할 수 있습니까?** 설명한다면 첫 문장은?"

### 분해 예시 — "고객 말만 들어도 진짜 문제를 안다"

| 프로브 | 답변 (예) | 변환 |
|--------|-----------|------|
| 어떤 표현이 신호인가 | "'그냥 좀 불편해서요'라고 얼버무리면 진짜 이유는 딴 데 있다" | `signals` |
| 말 vs 행동 | "말보다 '이미 뭘 해봤는지'를 믿는다. 해본 게 없으면 아직 안 아픈 것" | `rule` |
| 초보가 놓치는 것 | "신입은 요구사항 문장을 그대로 받아적는다" | `noviceMistake` |
| 빗나가는 경우 | "기술 배경이 있는 고객은 오히려 정확하게 말한다" | `exceptions` |
| 설명 가능한가 | "가능. '요청 뒤에 있었던 사건을 먼저 물어라'" | 규칙 문장 완성 |

이렇게 완성된 규칙 1개가 `judgmentRules[]`의 1개 항목이 된다.

## 3. Gate 4 질문 뱅크 — 4범주

### ① 반복 노출
- "가장 많이 반복한 일은 무엇입니까? 대략 몇 번(몇 년)입니까?"
- "남들보다 압도적으로 많은 사례를 본 분야는 무엇입니까?"

### ② 판단 기준
- "좋은 결과와 나쁜 결과를 무엇으로 구분합니까?"
- "문제가 생기기 전에 미리 알려주는 신호는 무엇입니까?"
- "매뉴얼에는 없지만 실제로 중요한 것은 무엇입니까?"

### ③ 예외
- "일반적인 방법이 안 통하는 경우는 언제입니까?"
- "초보자가 가장 자주 틀리는 지점은 어디입니까?"
- "규칙을 그대로 적용하면 오히려 문제가 되는 상황이 있습니까?"

### ④ 타인의 요청
- "사람들이 무엇을 들고 당신을 찾아옵니까?"
- "무료로도 반복해서 도와주고 있는 일은 무엇입니까?"
- "설명하지 않아도 주변이 인정해 주는 능력은 무엇입니까?"

①②③은 `judgmentRules[]`의 재료, ④는 `assets[]`(특히 `kind:"trust"|"relationship"`)와
수요 증거(`ledger.facts`)의 재료다.

## 4. 관심 vs 전문성 구분

"좋아한다·관심 있다·공부 중이다"는 전문성이 아니다. 아래 6종 증거 중 몇 개가 성립하는지 확인한다:

1. **반복 수행** — 실제로 여러 번 했다 (횟수·기간을 물어 확인)
2. **예외 해결** — 정석이 안 통하는 상황을 뚫어봤다
3. **판단 기준 설명 능력** — 왜 그렇게 하는지 남에게 설명할 수 있다
4. **타인에게 결과 제공** — 본인 아닌 타인의 문제를 해결해 줬다
5. **실패·교정** — 틀려본 적이 있고, 그 후 방식을 고쳤다
6. **타인의 신뢰·지불** — 남이 시간이나 돈을 내고 그 판단을 샀다

`confidence` 판정:
- `experience` — 위 증거 3개 이상, 각각 구체 사례 있음
- `talent` — 사례는 있으나 반복·지불 증거가 얇음 (재능은 보이지만 시장 검증 전)
- `unverified` — 본인 주장뿐. 이 규칙에 기반한 기회는 evidenceGrade D를 넘을 수 없다

관심에 불과한 영역이 발견되면 정직하게 말하고, 부족한 증거는 `ledger.gaps`에 남긴다.

## 5. 기록 형식

### judgmentRules[] — Gate 4 통과 조건: 규칙 3개 이상 + 각 규칙에 사례 필수 + 예외 1개 이상

```json
{ "id": "R1",
  "rule": "고객 요청 뒤에 있었던 사건을 먼저 물어라 — 요청 문장 자체는 진짜 문제가 아니다",
  "signals": ["'그냥 좀 불편해서요' 류의 얼버무림", "이미 해본 시도가 없음"],
  "example": "2025-11 A사 — '대시보드 요청'을 파니 실제 문제는 주간보고 취합 3시간이었다",
  "exceptions": ["기술 배경이 있는 고객은 요청이 곧 문제인 경우가 많다"],
  "noviceMistake": "요구사항 문장을 그대로 받아적고 견적을 낸다",
  "evidence": "동일 패턴 상담 30건 이상, 그중 재계약 12건",
  "confidence": "experience" }
```

- `example` 없는 규칙은 등록하지 않는다 — 사례를 먼저 받아낸다.
- `exceptions`가 하나도 없으면 "이 규칙이 안 통하는 경우는?"을 반드시 묻는다. 예외를 아는 것이 전문성의 표식이다.

### assets[] — 비대칭 자산 (Gate 4에서 발굴, Gate 6에서 보강)

```json
{ "id": "A1", "kind": "data",
  "description": "5년치 학원 상담 기록 스프레드시트 약 1,200행",
  "evidence": "2021년부터 누적, 실제 파일 존재 확인",
  "replicability": "hard" }
```

- `kind`: `experience` | `data` | `materials` | `relationship` | `trust` | `skill`
- `replicability`: 경쟁자가 이것을 확보하는 난이도 — `hard`(수년 필요) / `medium` / `easy`(누구나 곧 확보).
  `easy`인 자산은 기회의 `whyYou` 근거로 쓰지 않는다.

## 6. 그래프 연결 (Gate 5 준비)

mining에서 확보한 항목은 `graph.nodes`의 재료가 된다 — `type:"JudgmentSignal"|"DomainRule"|"Exception"|"DomainAsset"` 노드로 올리고,
`edges`로 `Experience → REVEALS → DomainRule`, `DomainRule → HAS_EXCEPTION → Exception` 관계를 잇는다.
노드에는 `evidenceIds`로 F-ID를 연결한다. 증거 없는 노드는 `status:"needs_evidence"`로 올린다.

## state 기록 요약

| 무엇을 | 어느 필드에 |
|--------|-------------|
| 광산에서 나온 실제 사건 | `events[]` |
| 반복 확인된 경험 사실 | `ledger.facts` (F-ID) |
| 분해 완료된 판단 규칙 | `judgmentRules[]` (`rule`, `signals`, `example`, `exceptions`, `noviceMistake`, `evidence`, `confidence`) |
| 관심/전문성 판정 | `judgmentRules[].confidence` (`experience`\|`talent`\|`unverified`) |
| 축적 자료·관계·신뢰·기술 | `assets[]` (`kind`, `description`, `evidence`, `replicability`) |
| 증거가 얇은 영역 | `ledger.gaps` |
| 규칙·자산의 그래프 반영 | `graph.nodes` (`type`, `status`, `evidenceIds`) + `graph.edges` |
| 채굴 질문 (1개씩, 이유 포함) | `currentQuestion` (`text`, `why`, `gate:"mining"`) |
